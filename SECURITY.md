# Security Policy

## Reporting a vulnerability

**Please do not open a public issue for security problems.**

Report vulnerabilities privately through GitHub's
[**private vulnerability reporting**](https://github.com/nickozz714/ND3X/security/advisories/new)
(the *Security* tab → *Report a vulnerability*). This keeps the report confidential
until a fix is available.

Please include:

- what the issue is and its impact,
- steps to reproduce (a proof of concept if possible),
- affected version / commit.

You'll get an acknowledgement, and we'll work on a fix and coordinate disclosure with you.

## Handling secrets (for operators)

ND3X is built so secrets never live in the repository:

- **No secrets in git.** There is no committed `.env`; `.env.example` ships with
  empty values only. `*.env`, `*.db`, and `*.sqlite*` are git-ignored.
- **Encrypted at rest.** Provider API keys, mail credentials and KeyVault secrets
  are stored **Fernet-encrypted** in the database (see `utils/crypto.py`), keyed by
  `MAIL_SECRET_KEY`. The plaintext is never returned by the API — reads expose
  metadata and an obfuscated value only.
- **JWT + Fernet keys** are generated by the first-run setup wizard and persisted
  under the base directory (default `~/.nd3x`), **outside** the repository.
- **Workflow secrets** (`${secret.NAME}`) are decrypted only server-side, at the
  outbound boundary, and masked in traces/output so the model never sees them.

When self-hosting, set `JWT_SECRET` and `MAIL_SECRET_KEY` to strong, unique values
and keep the base directory off any shared/public storage.
